Autor: Raúl Rodríguez · 2026

EFBY Request Lab1 / 18
Contexto

El problema

  • Los equipos dependen de clientes cloud (Postman, Insomnia) para probar APIs.
  • Es difícil versionar colecciones junto al código en Git sin fricción.
  • El Collection Runner es lineal: sin ramas condicionales ni paralelismo visual.
  • Falta una herramienta nativa macOS/iPad integrada en el ecosistema Swift del equipo.
EFBY Request Lab2 / 18
Propuesta

La solución

EFBY Request Lab es un laboratorio API nativo que ofrece:

  • HTTP y WebSocket con variables por capas
  • Scripts JavaScript compatibles con pm.*
  • Import/export Postman y OpenAPI
  • Workspace local + sync Git manual
  • Flujos BPMN con gateways y paralelismo

Usuarios objetivo

  • Desarrollador backend — endpoints y scripts
  • QA / integración — colecciones y flujos
  • DevOps — colecciones en repo Git
  • Equipo EFBY — herramienta offline

Arquitectura

Clean Architecture en capas SPM

EfbyRequestLabs / EfbyPostmanPad (UI SwiftUI) ↓ EfbyPresentation (MainViewModel, coordinadores, AppDependencies) ↓ EfbyApplication (Ports + Use Cases) ↓ EfbyDomain (modelos) ↑ EfbyInfrastructure (implementa Ports)

Domain no importa SwiftUI. Application no usa URLSession directamente: solo vía ports en Infrastructure.

EFBY Request Lab9 / 18
Flujo

Ejecución HTTP (ejemplo)

  1. La vista llama a MainViewModel.sendCurrentRequest().
  2. RequestTabCoordinator invoca ExecuteHTTPRequestUseCase.
  3. El use case usa HTTPExecutionServiceProtocol.
  4. RequestExecutionService ejecuta la petición y los scripts.
  5. El ViewModel aplica variables y persiste vía WorkspacePersistenceCoordinator.
EFBY Request Lab10 / 18
Persistencia

Tres capas de datos

1. Local

WorkspaceRepository guarda el workspace en Application Support: pestañas, historial, configuración.

2. Workdir Git

SharedCollectionsRepository gestiona colecciones, entornos y flows en el directorio compartido.

3. Remoto

Pull/push manual con GitSessionCoordinator y GitWorkspaceCoordinator. Sin backend propio ni sync cloud.

EFBY Request Lab11 / 18
Metodología

Specification-Driven Development (SDD)

Las specs son la fuente de verdad funcional antes de implementar el código.

  • 10 especificaciones en SDD/tile/specs/*.spec.md
  • Cada spec define comportamiento, criterios de verificación y trazabilidad
  • IDs REQ-* enlazados a tests en la matriz de trazabilidad
  • Reglas de desarrollo y seguridad en SDD/tile/rules/
